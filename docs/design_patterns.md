# Design Patterns Applied

Every pattern below is verified against the actual code â€” file and, where a single class/function anchors it, the line. This is not an aspirational list; each entry is grep-able.

| Pattern | Where it lives | Why |
|---|---|---|
| **Singleton** | `app/config.py:83` â€” module-level `settings = Settings()` | One `Settings` object, loaded once from `.env`, imported everywhere as `from app.config import settings`. Avoids re-reading/re-validating `.env` on every import and gives every module the same view of config. |
| **Factory** | `app/llm/factory.py:81,100` (`get_llm()`, `get_embeddings()`); `app/indicators/__init__.py:17` (`IndicatorFactory`); `app/memory/checkpointer.py:21` (`get_checkpointer()`) | Swap model tier / indicator / memory backend via config or a parameter, with zero changes at call sites. `get_llm(reasoning=True)` picks the reasoning-tier Gemini model instead of the worker tier; `get_checkpointer()` returns `SqliteSaver` today and `AsyncPostgresSaver` in production (see `production_plan.md`) behind the same signature. |
| **Template Method** | `app/agents/base.py:36` (`BaseAgent.run` = `_preprocess` â†’ `_invoke` â†’ `_postprocess`) | All 4 specialist agents share identical pre/post-processing (logging context, timing, persisting `tool_results`, surfacing RAG snippets into `retrieved_context`); each subclass overrides only `_invoke()`. Adding a 5th agent means implementing one method, not re-deriving the plumbing. |
| **Strategy** | Routing: `app/graph/router.py:36` (`RoutingStrategy` â†’ `LLMRoutingStrategy`/`KeywordRoutingStrategy`). | **Repository** | `app/data/repositories.py:34,42` (`PortfolioRepository`, `ClientProfileRepository` â†’ Excel-backed implementations); `app/knowledge/vector_store.py:29` (`VectorStore` wraps Chroma) | Agents/tools depend on the abstract repository, not on `pandas`/`openpyxl`/Chroma directly. `ExcelPortfolioRepository` unions two `.xlsx` files today; swapping to Postgres later (`production_plan.md`) is one new class implementing the same 2-method interface â€” zero agent changes. |
| **Adapter** | `app/integrations/base.py:17` (`MarketDataAdapter`) with 3 implementations (`finnhub_adapter.py`, `alpha_vantage_adapter.py`, `yfinance_adapter.py`) + `sec_edgar_adapter.py`, `news_adapter.py` | Every external market-data vendor is wrapped behind one interface (`get_quote`/`get_price_history`/`get_fundamentals`/`get_news`), each translating that vendor's own failure quirks (429s, silent "Note" fields, paid-plan errors) into this project's own exceptions. | **Registry** | `app/tools/registry.py:24` (`ToolRegistry`, singleton via `__new__`) | Tools declare themselves at their point of definition (`@tool_registry.register(agent="risk")`) and are discovered centrally (`tool_registry.tools_for("risk")`, `tool_registry.table()`). No hand-maintained "list of all tools" to fall out of sync â€” verified live in `architecture.md` Â§5's tool tables, which are the registry's own `table()` output. |
| **Decorator** | `app/tools/decorators.py:37,72,98` (`@cached`, `@retry`, `@rate_limited`) | Cross-cutting resilience concerns (TTL caching, exponential-backoff retry, token-bucket rate limiting â€” including throttling Gemini itself via `InMemoryRateLimiter` in `llm/factory.py:92`) applied by stacking decorators, without polluting any tool's own logic. Stack order (`cached â†’ retry â†’ rate_limited â†’ fn`) is deliberate: a cache hit must cost nothing downstream. |
| **Chain of Responsibility** | Guardrail pipeline: `app/guardrails/base.py:41` (`GuardrailPipeline.run`, short-circuits on first non-pass). | **Circuit Breaker** | `app/errors/circuit_breaker.py:37` (`CircuitBreaker`, 3-state: closed/open/half-open); `:101` (`CircuitBreakerRegistry`, one breaker per adapter name) | Turns "3 slow, doomed retries per call" into an instant fail-fast once an adapter has proven itself down for `CIRCUIT_BREAKER_THRESHOLD` consecutive failures, recovering automatically after `CIRCUIT_BREAKER_COOLDOWN` seconds via a half-open trial. Distinct from `@retry`, which only survives a single transient blip. |
| **Fallback** | `app/errors/fallback.py:18` (`Fallback.run`: primary â†’ secondary on ANY exception) | General-purpose primary/secondary pairing used outside the adapter chain â€” e.g. `Retriever.query()` (`app/knowledge/retriever.py:46`) falls from Chroma to an on-disk keyword search when the vector store is unreachable, so citations keep flowing (degraded ranking) instead of the tool erroring outright. |
| **Builder** | `app/graph/builder.py:55` (`GraphBuilder`, fluent `.with_supervisor().with_memory().with_planning()...with_all().build()`) | Progressive, phased assembly of the LangGraph â€” each `.with_X()` call is one optional stage (supervisor, memory, planning, guardrails, clarification), so tests can build a lean 1-agent graph while the CLI/UI use `.with_all()`. The `build()` method itself contains the conditional wiring logic (static edges vs `Command`-only routing, checkpointer injection) that every caller is shielded from. |
| **Supervisor / Multi-Agent orchestration** | `app/agents/supervisor.py:36` (`SupervisorAgent`) | The canonical LangGraph supervisor pattern: one node is the sole scheduler, specialists never call each other, and all coordination flows through `AgentState` + `Command(goto=...)`. See `architecture.md` Â§4 for the full protocol writeup. |
| **Command** | Planner output: `app/graph/planner.py` (`plan: [{"agent","goal"}]`); executed by `SupervisorAgent.run` (`app/agents/supervisor.py:57`) | Each plan step is a small, inspectable, replayable instruction object â€” not an inline branch of control flow. `state["plan"]` is a complete, structured record of what was decided and why, recoverable from any checkpoint (visible directly in `docs/evaluation_report.md`'s traces and the CLI's live plan printout). |
| **State (formal state machine)** | LangGraph's own `StateGraph` + `interrupt()`/`Command(resume=...)` (`app/graph/clarifier.py:73`) | "Waiting for human input" is a first-class graph state (persisted via the checkpointer), not a bolted-on exception or a separate polling loop. The graph's execution model *is* a state machine; `interrupt()` pauses it mid-transition and `Command(resume=...)` is a callback that resumes exactly where it left off, even across process restarts (backed by SQLite). |
| **Producer/Critic (Reflection)** | `app/graph/synthesizer.py:39` (`SynthesizerNode`, the producer) â†” `app/graph/reflector.py:76` (`ReflectorNode`, the critic) | The reflector audits the synthesizer's answer against `tool_results`/`retrieved_context` via a 4-guard pipeline and, on a fixable problem, loops back with a specific critique (`"REVISION REQUIRED: â€¦"`) rather than a generic "try again" â€” caught two real hallucinated sector classifications live during Phase 14 verification (see `architecture.md` Â§6). |
| **Interpreter** | `app/graph/planner.py` (produces the plan) executed by `app/agents/supervisor.py:52` (interprets it step by step) | The plan is a tiny domain-specific language â€” `[{"agent": str, "goal": str}, ...]` â€” and the supervisor is its interpreter: advance `plan_step`, dispatch `plan[plan_step]["agent"]` with a focused instruction, repeat until exhausted. |
| **Facade** | `app/knowledge/retriever.py:23` (`Retriever.query()` hides Gemini embedding + Chroma search + metadata filtering + fallback behind one call); `app/memory/store.py:45` (`MemoryStore` hides SQLite behind `save_decision`/`get_recent_decisions`); `app/graph/builder.py`'s `.build()` (UI-facing) | Each hides a genuinely complex subsystem behind a single, simple call. `ui/streamlit_app.py` never learns Chroma, SQLite, or LangGraph's node wiring exist â€” it calls `GraphBuilder().with_all(checkpointer=...).build()` and streams events. |
| **Observer** | LangSmith tracing (`app/config.py` â€” activates `LANGCHAIN_TRACING_V2` when a key is present; every LangChain/LangGraph call auto-emits trace spans with no code-level coupling); streaming event handling (`ui/streamlit_app.py`'s `stream_turn`, the CLI's `graph.stream(..., stream_mode="updates")` loop) | Telemetry and UI updates observe the graph's execution without the graph knowing they exist â€” removing LangSmith or the UI's event collector changes nothing about how a node runs. |
| **Interceptor / Middleware** | `app/guardrails/access_control.py:39` (`verify_client_access`, called from every repository method); `app/errors/node_wrapper.py:52,69` (`wrap_dict_node`/`wrap_command_node`, wraps every graph node when guardrails are on) | Cross-cutting policies applied uniformly at a single chokepoint rather than duplicated per call site. `verify_client_access` sits inside `ExcelPortfolioRepository.get()` itself, so **every** current and future client-scoped tool is covered automatically, even a prompt-injected "show me CLT-001's data" from inside a CLT-002 session â€” it dies at the repository, regardless of how convincingly the LLM was tricked into calling the tool. The node wrapper does the same for uncaught exceptions: no stack trace ever reaches the user (Phase 11's explicit requirement), while still letting `GraphInterrupt` (LangGraph's own pause signal) propagate untouched. |

## Notes on two deliberately-scoped exceptions

- **Optimization objectives (Strategy)** â€” `app/tools/risk_tools.py`'s VaR strategies and the indicator Strategy family are fully built and tested; the Optimization Agent itself (Phase 12, "efficient frontier, objective functions as Strategy") is scaffolded (`notebooks/phase12_optimization.ipynb`) but marked **bonus/not yet implemented** per the documented scope cut line in `docs/00_prerequisites_and_setup.md` Â§8. The Strategy pattern shape for it (one `ObjectiveFunction` interface, multiple implementations selected by name) would mirror `VaRStrategy` exactly if built.
- **Interceptor is used twice, deliberately for different concerns** â€” `access_control.py` is the *data*-privacy interceptor (client boundary enforcement); `node_wrapper.py` is the *execution*-safety interceptor (no raw errors reach the user). They compose independently and neither depends on the other.
